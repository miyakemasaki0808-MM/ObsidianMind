# 関連ノートAI推薦の設計

ノートを開いたとき、話題的に近い他ノートを提示する機能の設計と、その改修（Phase 1〜3）で得た知見をまとめる。

- **今どうなっているか**は [source_code_analysis.md](../source_code_analysis.md)、**いつ変えたか**は [change_history.md](../change_history.md) を参照。
- 対象コード: `domain/RelatedNotesUseCase.kt` ほか `domain/RelatedCandidate*.kt` / `RelatedContextScoring.kt`。

---

## 1. 背景と課題

Obsidian Vault は数百〜のノートを持ち、多くが `0F01 タイトル` のような**16進採番プレフィックス**を持つ。当初の関連ノートは「採番が近い（上位桁が一致する）ノート」を出すだけで、以下の課題があった。

1. **採番偏重**: 「関連」が *話題が近い* ではなく *採番が近い* になっていた。話題は近いが採番の離れたノートは一切出てこない。
2. **タイトルしか見ていない**: AIに候補タイトルだけを渡していたため、曖昧なタイトルの判断材料が乏しい。
3. **長いノートの文脈が伝わらない**: 先頭だけを見て要約・推薦していた。
4. **AI応答の解決が不安定**: モデルにタイトル文字列を返させると、言い換え・翻訳・装飾・同名衝突で解決に失敗した。

これらを Phase 1〜3 で段階的に解いた。オンデバイス（Gemini Nano / ML Kit GenAI）前提のため、**小さいモデルの制約（出力トークン上限・入力予算）** と **SAF（別プロセス）の I/O コスト** が全設計を貫く制約になっている。

---

## 2. 全体アーキテクチャ — 2チャンネル

関連ノートは性質の異なる2チャンネルで提示する。**役割を分離**することで、AIチャンネルを「まだ表示していないノートの補完」に純化できる。

| チャンネル | 中身 | 実装 |
|---|---|---|
| **決定的チャンネル** | wikilink 済み ＋ 採番上位2桁一致のノート（最大5件） | `buildDeterministicRelatedNotes` |
| **AIチャンネル** | 上記を除いた候補を Nano に選ばせた5件 | AIブランチ（下記パイプライン） |

AIチャンネルのパイプライン（Phase 3 完成形）:

```
全Vaultのノート
  │  ① タイトル話題スコアで全件ランク（Phase 3a）→ 上位40件
  ▼
上位40件
  │  ② 本文を上限付き並列で読込・キャッシュ（Phase 2b）→ snippet/tags/aliases
  ▼
40件＋本文
  │  ③ 現ノートの本文シグナルで再ランク（Phase 3b）→ 並べ替えのみ
  ▼
再ランク済み40件
  │  ④ 一時ID(C01..)採番 → 入力予算内へ整形（Phase 2a）→ プロンプト
  ▼
Nano
  │  ⑤ IDのみ応答 → ノートへ解決（Phase 1b）
  ▼
関連5件
```

**設計の背骨**: 純ロジック（`android.net.Uri` 非依存・`internal`）と I/O（SAF読込・Nano生成）を分離し、前者を JVM ユニットテストで完全にカバーする。並べ替え・スコア・整形・ID解決はすべて純ロジック側にある。

## 3. フェーズ別の設計判断

### Phase 1 — 正確性とID応答方式（PR #27）
- **バブル（フォーカス中セクション）文脈**で推薦し、長いノートの先頭切り出し問題を解消。
- **ID応答方式**: 候補に一時ID `C01..` を採番し、Nano には**IDだけ返させる**（`RelatedCandidateId.kt`）。タイトル文字列を返させると言い換え・翻訳・装飾・同名衝突で解決が壊れる。IDなら1トークン級で確実に解決でき、出力トークンも節約できる。

### Phase 2 — 情報強化・キャッシュ・並列（PR #28）
- **候補の肉付け（2a）**: タイトルに加え本文冒頭スニペット・タグ・aliases を渡す（`RelatedCandidateContext.kt`）。プロンプト全体が膨らみすぎないよう、**入力予算内へ動的に短縮**（タグ→aliases→スニペットの順に削る／ID・タイトルは常に残す）。
- **キャッシュ（2b）**: 候補本文を `URI+lastModified` キーでメモ化（`KeyedMemoCache.kt`）。**成功時のみ格納**し、失敗・キャンセルはキャッシュしない（次回リトライ）。Vault切替時に破棄。
- **上限付き並列読込（2b）**: `Semaphore(8)` で SAF 同時アクセス数を絞りつつ `async/awaitAll`。順序は awaitAll のリスト順で保証、キャンセルは `coroutineScope` で構造化。

### Phase 3a — 全Vaultタイトルスコアリング（PR #29）
採番順の並べ替えを、**タイトルの話題類似スコア**による全Vaultランキングへ置換（`RelatedCandidateScoring.kt`）。

```
score = W_TITLE * titleSim(現, 候補) + W_PREFIX * prefixTier(現, 候補)
```

- `titleSim` = 採番除去・正規化したタイトルの **文字bigram Dice係数**。
- `prefixTier` = 採番の 2桁一致=1.0 / 1桁一致=0.5 / それ以外=0.0。
- 話題類似を主、採番近接を弱い従シグナルに。→ 採番が離れても話題が近ければ上位に来る。

### Phase 3b — 本文シグナルによる二段ランク（PR #29）
Phase 2b で読み込んだ上位40件の本文を使い、**プロンプト直前に再ランク**（`RelatedContextScoring.kt`）。件数は変えず並べ替えのみ。

```
score = W_TAG*tagDice + W_BODY*snippetDice + W_TITLE_SIM*titleDice
tieBreak = titleDice
```

- タグ一致を主シグナルに、本文スニペット類似・タイトル類似を弱く加点。
- tags 無し候補は本文・タイトルへ自然にフォールバック。
- 目的は **Nano の位置バイアス対策** — プロンプト先頭に話題的に近い候補を寄せる。

## 4. 得られた知見（このプロジェクト固有の学び）

### 4.1 日本語は「文字bigram Dice」でトークナイザなしに類似が測れる
日本語タイトルは空白で単語分割できない。形態素解析器を持ち込まず、**文字2-gram集合の Dice 係数**で類似を測ると、日本語・英語・ローマ字が同じ式で扱え、純ロジックのまま JVM テストできる。追加ライブラリもゼロ。短い文字列（タイトル・スニペット）同士の比較に特に相性が良い。

### 4.2 「役割の分離」がAIチャンネルの質を上げた
決定的チャンネル（確実な関連）とAIチャンネル（補完）を分け、**AIには決定的枠を除外した候補だけ**渡す。これで上限5枠を既出ノートで浪費せず、AIを「未表示ノートを掘る」用途に純化できた。

### 4.3 モデルにはIDを返させる（テキストを返させない）
小さいモデルほど、自由記述の返答は表記ゆれで壊れる。**選択肢にIDを振って番号で返させる**設計は、解決の確実性・出力トークン・パースの頑健性すべてで有利。行頭付近のIDのみ拾い、桁落ち・大小文字を吸収する（`parseCandidateIds`）。

### 4.4 キャッシュは「成功だけ」保存し、キャンセルを汚さない
`getOrLoad` は load 成功時のみ格納。例外（キャンセル含む）は何も残さず伝播 → 途中結果を確定キャッシュにしない。ノート切替の頻繁なキャンセルがあっても、キャッシュ整合が壊れない。

### 4.5 SAFは別プロセス — 並列は「上限付き」で
候補40件を無制限に並列読込すると SAF プロバイダが詰まる。`Semaphore(8)` で同時数を絞ると、速度を稼ぎつつ詰まりを避けられた。順序は `awaitAll` のリスト順保証に委ねる。

### 4.6 二段ランク＋位置バイアス
全件はタイトルで安く粗選別（1段目）、上位だけ本文を読んで精選別（2段目）。**本文読み込みは上位40件のみ**でコストを据え置きつつ、プロンプトの並び順を話題順にして Nano の選択品質を底上げした。

### 4.7 「採点」と「並べ替え」の分離が投資として効いた
Phase 3b の前に、ランキングを **採点戦略（`scoreOf`）を注入する汎用パイプライン（`rankByScore`）** へリファクタした（挙動不変の準備コミット）。おかげで Phase 3b は「本文シグナルの `scoreOf` を1本書いて差し替える」だけで済み、除外・並び順・タイブレーク・上限のロジックを重複なく共有できた。**"差し込みやすい土台"を先に作る**判断が、本体実装を最小化した。

### 4.8 実データでの手応え
Phase 3b 適用後、`jetpackcompose` ノートの関連候補が **すべて Kotlin 系ノートで揃った**。採番だけを見ていた頃には出てこなかった、話題的に一貫した並び。タグ・本文シグナルが期待どおり効いていることの実機確認になった。

### 4.9 定数はすべて「実機計測で調整する前提の初期値」
重み・長さ・予算・キャッシュ件数・並列数はコード1か所の定数に集約し、コメントで「調整前提」と明記。実データを見てから1行で動かせるようにしておく。
