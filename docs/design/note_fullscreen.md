# 設計思想 — 全画面ノート（独立ルート化）

**対象領域:** 全画面ノート閲覧・システムバー・スクロール継承
**初版:** 2026-07-21（PR #30）
**関連:** [tab_navigation](tab_navigation.md)（全画面表示の前身＝オーバーレイ方式）

---

## 解決したかった問題

「Rediscover（ノートタブ）で開いたノートを、折り畳み前後どちらでも“ノート以外”を完全に消して読みたい」。しかし従来の全画面はこれを満たせなかった。原因は1つではなく4層に分かれていた。

- **アプリのバー/レールが消えない（本丸）**: 全画面は `AppScaffold` の `content` 内に置いたオーバーレイだった。compactでは下バーと本文が別領域（`Column`）、expandedでは左レールと本文が別領域（`Row`）に分かれているため、本文内のオーバーレイでは構造的にバー/レールを覆えない。
- **システムバーが残る**: グローバルの `hideStatusBar()` は `statusBars()` しか隠しておらず、ナビゲーションバーは対象外。
- **額縁が残る**: オーバーレイが `ReadingGradient` ＋ `safeDrawingPadding` ＋ 12dp paddingを持ち、ノートパネルの外周にグラデの額縁が見えていた。
- **スクロールが先頭に戻る**: オーバーレイ側の本文パネルに `listState` を渡していなかったため、全画面を開くたびに独立した状態で先頭から表示されていた。

## 判断1: オーバーレイをやめ、非タブの独立ルートにする

`note_fullscreen` を NavHost の非タブルートとして追加した。`AppScaffold` は既に「非タブルート（quiz/annotation 等）ではバー/レールを出さない」構造を持つため、ルート化するだけで**折り畳み前後どちらでもバー/レールが自動的に消える**。`isFullscreen` の `remember` フラグと違い、ルートは NavController の復元対象になるため、**Fold開閉によるActivity再生成でも全画面が維持される**。

前身の [tab_navigation](tab_navigation.md) 判断2は「⛶で開く／✕で閉じる明示操作」を残しつつ、その表示レイヤーをオーバーレイからルートへ移した形になる。

## 判断2: システムバーはルートの生存期間に紐づけて制御する

`FullscreenNoteScreen` の `DisposableEffect` で、進入時に `systemBars()`（ナビ＋ステータス）を隠し、離脱時に **`navigationBars()` だけを復元**する。ステータスバーはアプリ全体仕様どおり隠したままにし、既存の `onWindowFocusChanged → hideStatusBar()` と矛盾しない。復元をライフサイクルに紐づけることで、✕・システムバック・プロセス復元など全ての離脱経路で確実に戻る。

## 判断3: スクロールは「共有」ではなく「継承＋書き戻し」

当初は通常表示と全画面で単一の `LazyListState` を共有する案だった。しかし **NavHost の遷移アニメーション中は通常タブと全画面が同時にコンポーズされる**ため、1つの `LazyListState` を2つの `LazyColumn` に装着すると例外になる。

そこで全画面は専用の `LazyListState` を持ち、

1. 進入時にタブ側の `firstVisibleItemIndex`/`ScrollOffset` から**開始位置を継承**（読み取りのみなので同時装着は起きない）
2. 離脱時（✕・システムバック・FABタップ）にタブ側へ**書き戻す**（`BackHandler` でシステムバックも同期）

とした。これで双方向にスクロール位置が保たれる。`rememberLazyListState(index, offset)` は内部で `rememberSaveable` を使うため、全画面中のFold開閉でも位置が復元される。

> **書き戻しの方式変更（PR #31）**: 当初は離脱時に suspend の `scrollToItem` の完了を待ってから閉じる処理（`popBackStack`）を呼んでいた（フリング中に書き戻しがpop→scopeキャンセルで消えるのを防ぐ狙い）。しかしこれは「閉じる」を「完了保証のない suspend」に直列依存させる形で、**Fold開閉によるActivity再生成後に `scrollToItem` が完了せず、✕もシステムバックも無反応で全画面を解除できない**不具合を生んだ。現在は非suspendの `requestScrollToItem` で保留位置を積むだけにし、閉じる処理を即実行して書き戻しから切り離している（コルーチンを介さないため元の「書き戻し消失」も原理的に起きない）。教訓は [bugfix_reports.md](../bugfix_reports.md) #2 を参照。

## 判断4: ノート色で全ブリードし、本文カラムだけ720dpに絞る

背景を白カード＋グラデ額縁ではなく、**画面全体をノートページ色で塗り、本文カラムを最大720dpで中央寄せ**にした。安全インセット（`safeDrawingPadding`）は背景ではなく本文・✕ボタン側にだけ適用する。これにより広い展開画面でも行長を抑えつつ、「ノート以外の額縁」を排除できる。

## 判断5: 全画面のAI導線は「消さず、最小化する」

全画面中もAIの状態が気になる（要約は終わったか、クイズは生成できたか）というユースケースを尊重し、AIインジケータFABは残した。ただし通常FABの立体グラス（62dp・影・グラデ）は使わず、**44dpのフラット円**に置き換える。

- 状態は円の中身（スピナー／✓／!／💬）で表現し、**完了・エラーへ変わった瞬間だけ短いラベルを数秒フラッシュ**して自動で畳む
- ステータスは要約/回答（`SectionChatState` 由来）に**クイズ状態（`QuizState`）を合成**する。優先順位は「生成中 > 未確認エラー > 未確認完了 > 待機」
- タップは要約シートを開く。シートは note ルートで描画されるため、**全画面を閉じてからシートを開く**

あわせて、全画面ルート表示中は quiz/annotation の Snackbar を抑制した（`isFullscreenRoute` ガード）。「ノート以外を出さない」思想と、状態の気づきを最小FABへ一本化する狙いによる。

---

## 検証で確認した観点

- compact/expanded 両方で、進入時にバー/レール・システムバー・額縁が消え、スクロール位置が継承される
- 離脱時にナビバーが復帰し、ステータスバーは隠れたまま
- 全画面中のFold開閉で全画面が維持される
- 要約/クイズ完了時にFABがスピナー→✓＋ラベルフラッシュ
