## 解決したかった問題

Reflectの理想は「**自分の考えをもう一段深くする読書相手**」。蒸留（[reflect_distill](reflect_distill.md)）は"ノートの中身"を育てるが、**ノートをまたぐ思考の連続性**は扱えない。

このアプリの心臓は **Rediscover（ランダムに過去ノートを引く）**。しかし現状、再発見は「たまたま出てくる」だけで、そこに**意図の糸**がない。

タイムカプセルは、読み終わりに**未来の自分へ1問だけ残し**、後日 **Rediscover でそのノートが引かれた時に問いが蘇る**機能。ランダム再発見に「前回の自分からの申し送り」という意味を与え、再発見を"偶然"から"再会"へ変える。**このループを持つこのアプリでしか作れない**のが最大の狙い。

## 判断1: 残すのは「1問」。v1で自由記述の「反応」も残せる（草案から改訂）

- 読み終わりに、未来の自分への**問いを1つ**残す
- **蘇り（再会）時に、その問いへ自由記述の反応を残せる**（日付付き）。反応は積み重なり、**問い→反応→反応… の思考の糸**になる（草案の「答えを書かせない」を、ユーザー決定により改訂）
- 1ノートにつき**アクティブなカプセルは1つ**（問い＋その反応スレッド）。新しい問いを残すと置換（複数世代の履歴保持はv2）

## 判断2: AIは「叩き台を1つ」だけ、オンデマンドで提案

- 白紙から書いてもよい。「AIに叩き台をもらう」タップで**AIが問いを1つ提案**
- **自動では呼ばない**（`AICoreClient.generate()` は Mutex 直列＋数十秒。読み終わるたびに強制発火させない）
- 生成は短い1回のみ。失敗/タイムアウトは黙って手書きへフォールバック
- 蒸留（生成しない）と違い、**生成的AIの正当な使いどころ**。主役はユーザーの問いで、AIは種を出すだけ

## 判断3: 保存は vault内の隠しフォルダにサイドカー（本文は編集しない）

- カプセルはノート本文(.md)に埋め込まず**別ファイル**に持つ（frontmatter埋め込み＝本文編集の truncate窓リスクを避ける）
- 置き場所は **vault内の `.`始まり隠しフォルダ**（例 `.obsidianmind/timecapsules/`）。Obsidianは `.` 始まりを非表示にするため**vaultを視覚的に汚さず、かつ同期で別端末・再インストールをまたいで生存**する
- `_AI補記`（ユーザーが読む機能成果物・`AnnotationManagerScreen` が全削除）とは**用途が違う**ため使わない
- **1カプセル=1ファイル**（全体を毎回書き直す索引ファイルの破損・競合を避ける）。ファイル名はキー（後述）から導出（安全化のためハッシュ等）
- **書き込みは軽量な原子更新**（temp→rename＋チェックサム）。反応追記でファイルを更新するため。※ただし対象はアプリのメタデータであり、破損しても失うのはカプセルであってユーザーのノートではない（蒸留の元ノート書き込みより stakes は低い）

## 判断4: 可搬キーは「vault相対パス＋タイトル」。documentId は端末内キャッシュ（草案から精密化）

**SAF documentId は端末／権限グラントごとに異なる**ため、同期した別端末では同じファイルでも別IDになる＝**documentId は可搬キーにならない**。判断3が「同期で生き残る」ことを狙う以上、キーは可搬でなければならない。

- **可搬キー＝vault相対パス**（例 `ideas/habit.md`）を主キーにする（同期をまたいで安定）
- **タイトル**を副キー（表示＋再マッチ）に保持
- **documentId** は端末内の高速引き当てキャッシュに留める（グラント内で安定・別端末では無効）
- 引き当て順：documentId（端末内・速い）→ vault相対パス（可搬）→ **タイトル再マッチ**（ユーザー指定のフォールバック）
- **タイトル再マッチのあいまい性**：同名ノートが複数ある時は**自動で結び付けない**（誤マッチ防止）。一意に定まる時だけ再バインドしてパス/documentId を更新。あいまい時は孤立扱い（v2で「未回答カプセル一覧」から手動解決）

## 判断5: 蘇りは Rediscover 限定（草案から変更）

- カプセルのあるノートが **Rediscover（`loadRandomNote`）で引かれた時だけ**、本文冒頭に「前回の自分からの問い」カードを出す
- **検索・関連・直接オープンでは出さない**（"再会"感を濃くするユーザー選択）。意図の糸が濃くなる代わりに、目的を持って開いた時には出ない、という**意図的な割り切り**
- 実装上は「今回の表示が Rediscover 由来か」を state で区別し、その経路でのみカプセル照合する
- **残す**のは Reflect（AIタブ）の「タイムカプセルを残す」アクション

## 判断6: 通知はカードのみ

Rediscover で蘇った時は**本文冒頭のカード表示のみ**（Snackbar等は出さない）。カードは dismissible（「読んだ」で畳む／「反応を残す」／「新しい問いに置換」）。

## 仕様詳細（v1）

### データモデル（暫定）
```
TimeCapsule(
  vaultRelativePath: String,   // 可搬・主キー
  noteTitle: String,           // 副キー（再マッチ）＋表示
  documentId: String?,         // 端末内キャッシュ（可搬でない）
  question: String,            // ユーザーの問い（AIは叩き台のみ）
  responses: List<Response>,   // 再会時の自由記述（日付付き・積み重なる）
  createdAt: Long,
  sourceContentHash: String?   // 参考: 残した時点の本文（キーには使わない）
)
Response(text: String, createdAt: Long)
```

### フロー
1. **残す**：Reflectで「タイムカプセルを残す」→ 問いを入力（任意でAI叩き台1回）→ サイドカーへ原子保存。同一ノートの既存カプセルは置換
2. **蘇る**：Rediscover でノートが引かれる → 可搬キーで照合（判断4の順）→ ヒットなら冒頭カードに問い＋過去の反応を提示
3. **反応する**：カードから自由記述で反応を追記（日付付き）→ サイドカーを原子更新
4. カード操作：「読んだ」で畳む／「新しい問いに置換」

### AI設計
`AICoreClient` を流用。**オンデマンド1回のみ**の叩き台生成。失敗/タイムアウトは手書きへ静かにフォールバック。

### サイドカー書き込みの安全性
反応追記で更新が発生するため、**temp書き→rename＋チェックサム**で原子的に更新（[reflect_distill](reflect_distill.md) の `DistillRecoveryStore` 系パターンを流用）。破損検知時はそのカプセルを孤立扱いにし、ユーザーのノートには一切触れない。

## オープン論点（残り・実装前に詰める）
1. **サイドカーのファイル形式**：JSON か 独自バイナリか（可読性 vs 厳密性）
2. **タイトル再マッチのあいまい解決UI**：孤立カプセルをv1でどこまで見せるか（v2の一覧に寄せてよいか）
3. **反応スレッドの上限**：反応の件数・1件の文字数上限
4. **置換時の旧スレッド**：v1は破棄でよいか（履歴保持はv2）

## スコープ外（v2以降）
- 複数世代の履歴（問いの世代管理）
- 「未回答カプセル一覧」ビュー（Vault横断で"考えかけ"と孤立カプセルを俯瞰・手動解決）
- 蒸留（案A「読み直す観点」）との連携
- 反応の思考差分テンプレ（今も同意/変わった/新しい気づき＝Codex案2）

## レビュー観点（レビュー用）
1. **可搬キー設計**：vault相対パスを主キー・documentIdをキャッシュ・タイトル再マッチという三段が、同期・rename・move で破綻しないか
2. **Rediscover限定の判定**：「今回の表示が Rediscover 由来か」を state で確実に区別できるか（回転・プロセス復元をまたいでの誤判定はないか）
3. **サイドカー原子更新**：反応追記時の原子性・破損検知・孤立化の扱い
4. **反応スレッドのスコープ**：自由記述をv1に入れたことで肥大化しないか（上限・置換方針）
5. **AIオンデマンド1回**：負荷・失敗時フォールバック
6. **タイトル再マッチのあいまい性**：同名ノート時に誤バインドを避けられるか
