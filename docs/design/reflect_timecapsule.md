## 解決したかった問題

Reflectの理想は「**自分の考えをもう一段深くする読書相手**」。蒸留（[reflect_distill](reflect_distill.md)）は"ノートの中身"を育てるが、**ノートをまたぐ思考の連続性**は扱えない。

このアプリの心臓は **Rediscover（ランダムに過去ノートを引く）**。しかし現状、再発見は「たまたま出てくる」だけで、そこに**意図の糸**がない。

タイムカプセルは、読み終わりに**未来の自分へ1問だけ残し**、後日その問いが**ノート再訪時に蘇る**機能。Rediscoverのランダム性に「前回の自分からの申し送り」という意味を与え、再発見を"偶然"から"再会"へ変える。**この構成（ランダム再発見ループ）を持つこのアプリでしか作れない**のが最大の狙い。

## 判断1: 残すのは「1問」だけ（答えは求めない）

- 読み終わりに、未来の自分への**問いを1つ**だけ残す
- 答えを書かせない（負担最小・Codex案2の思考差分ログはv2）
- 1ノートにつき**アクティブなカプセルは1つ**。新しく残すと前のは置換（履歴保持はv2）

## 判断2: AIは「叩き台を1つ」だけ、オンデマンドで提案

- ユーザーが白紙から書いてもよいし、「AIに叩き台をもらう」をタップすると**AIが問いを1つ提案**する
- **自動では呼ばない**（`AICoreClient` の `generate()` は Mutex 直列＋数十秒。読み終わるたびに強制発火させない）
- 生成は短い1回のみ。プロンプトは「このノートを読み終えた人が未来の自分へ問いかけるとよい問いを1つ、短く」

これは蒸留（生成しない）と違い、**生成的AIの正当な使いどころ**。ただし主役はユーザーの問いで、AIは種を出すだけ。

## 判断3: カプセルはノート本文を編集せず、サイドカーに持つ

蒸留と同じくノート本文(.md)の破損リスクは避ける。カプセルは本文に埋め込まず**別データ**として保持する（frontmatter埋め込みは本文編集＝truncate窓リスクの再来なので避ける）。

保存先の候補は**オープン論点（後述）**。有力は「vault内サイドカー」か「アプリ専用ストレージ」。

## 判断4: ノート識別は documentId を基軸にする

カプセルを後日ノートへ結びつけるには**安定キー**が要る。蒸留レビューで判明した通り `NoteState.Success(title, content)` は**URIもハッシュも持たず**、タイトルは非一意。

- キーは **SAF documentId**（vaultツリー内でファイルに安定・move/renameで変わる点は許容範囲）を基軸
- タイトルは表示用＋ソフトフォールバック
- 本文ハッシュは**キーにしない**（編集で切れると再会できず本末転倒）。ただし「残した時点の本文か」を示す参考情報としては保持可

このため蒸留と共通で、**編集/照合対象を安全に特定するためのノート識別（URI＋documentId）を状態へ持たせる**必要がある（[reflect_distill](reflect_distill.md) 判断7と共有）。

## 判断5: 表示は「ノートを開いた時」、残すのは Reflect から

- **蘇り**：カプセルのあるノートを**開いた時**、本文冒頭に「前回の自分からの問い」カードを出す。ランダム再発見が情緒的な主動線だが、検索・関連から開いた時に取りこぼさないよう**"開いた時"全般で表示**する（Rediscoverは"なぜ戻るか"の演出）
- **残す**：Reflect（AIタブ）に「タイムカプセルを残す」アクション。読み終えて振り返る文脈に置く
- カードはドラッグ不要・** dismissible**。「読んだ」で畳む／「新しい問いを残す」で置換

## 仕様詳細（v1・暫定）

### データモデル（暫定）
```
TimeCapsule(
  noteDocumentId: String,   // 安定キー
  noteTitle: String,        // 表示・ソフトフォールバック
  question: String,         // ユーザーの問い（AIは叩き台のみ）
  createdAt: Long,
  sourceContentHash: String? // 参考: 残した時点の本文（キーには使わない）
)
```

### フロー
1. Reflectで「タイムカプセルを残す」→ 入力（任意で「AIに叩き台をもらう」1回）
2. 保存（サイドカー or アプリ領域：オープン論点）。同一ノートの既存カプセルは置換
3. 後日、そのノートを開く → documentId でカプセルを引き当て → 冒頭にカードで問いを提示
4. 「読んだ」で畳む／「新しい問いを残す」で置換

### AI設計
- `AICoreClient` を流用。**オンデマンド1回のみ**の生成（叩き台）。失敗/タイムアウトは黙って手書きにフォールバック

## オープン論点（確定前に決める）

1. **保存場所**：
   - 案a：**vault内サイドカー**（`_AI補記` とは別の隠しフォルダ等）。Obsidian同期で**別端末・再インストールをまたいで生き残る**（＝長期価値のある本機能に適合）が、vaultにファイルが増える＋documentIdベースの結びつけが要る
   - 案b：**アプリ専用ストレージ/DB**。vault非汚染で綺麗だが、**アンインストール・機種変で消える**（数ヶ月後の再会が売りなのに失われるのは痛い）
   - → 長期価値を重視すると案a寄りだが、vault footprint とのトレードオフ。**要決定**
2. **識別の安定性**：documentId が move/rename で変わった時の扱い（切れたカプセルの掃除 or タイトル再マッチ）
3. **蘇り条件**：本当に「開いた時」全般でよいか、Rediscover（ランダム）に限定して"再会"感を強めるか
4. **答えの記録**：v1は問いの提示だけ。将来 Codex案2（今も同意/変わった/新しい気づき）と接続するか＝v2判断
5. **通知**：カプセルのあるノートを引いた時、カード表示だけでよいか（Snackbar等は不要か）

## スコープ外（v2以降）
- 問いへの回答記録・思考差分ログ（Codex案2との統合）
- カプセルの履歴（複数世代）
- 「未回答カプセル一覧」ビュー（Vault横断で"考えかけ"を俯瞰）
- カプセル残し時の観点連動（案A「読み直す観点」との連携）

## レビュー観点（草案・Codex向け）
1. 保存場所（vault vs アプリ領域）の妥当性と、長期生存 vs vault footprint の判断
2. documentId 基軸の識別が move/rename/同期でどこまで安定か
3. ノート識別（URI/documentId）の状態保持を蒸留と共通化する設計でよいか
4. AIオンデマンド1回の生成方針（負荷・失敗時フォールバック）
5. 「開いた時に表示」のUXが煩わしくならないか（dismiss後の再表示ポリシー）
