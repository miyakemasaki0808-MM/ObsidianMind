# 設計思想 — セクション単位AIチャット

**対象領域:** ノート読書中のAI対話（浮遊吹き出し・ボトムシート）
**初版:** 2026-07-17（PR #14）

---

## なぜ「セクション単位」なのか

長いノート全体をオンデバイスのGemini Nanoに渡すと、コンテキスト長制限で後半が切り捨てられ、回答の的確さも落ちる。「今見ている部分だけ」を対象にすれば、長いノートでも入力が制限内に収まりやすく、回答の根拠も絞られる。

対象セクションはユーザーに選ばせず、スクロール位置から自動追従する。本文表示を `LazyColumn` 化して `firstVisibleItemIndex` を取得し、その直近の見出し（〜次の同レベル以下の見出し直前まで）を対象とする。見出しの無いノートはノート全体にフォールバックする。

## スコープの割り切り

| 論点 | 決定 | 理由 |
|------|------|------|
| 質問の入力方法 | 自由記述欄なし。AIが生成した質問候補のタップのみ | 入力の手間を省き、セクション範囲外の質問を構造的に防ぐ |
| 参照範囲 | 現在ノートの当該セクションのみ | 範囲外は「記載なし」と回答させ、推測による誤答を防ぐ |
| 会話履歴 | シートを開いている間だけ保持（揮発） | 永続化は保存先設計が必要になるため、ニーズが出るまで見送り |

## デザインの試行錯誤（6版の反復）

ユーザーフィードバックを受けて方向修正を重ねた。UIの初期案がそのまま採用されることはない、という好例。

| 版 | 内容 | 判断 |
|----|------|------|
| v1 | 見出しごとに個別チップを配置 | ユーザーイメージ（楽天のような浮遊ボタン）と不一致のため破棄 |
| v2 | 画面右下の浮遊吹き出し＋ドラッグ移動 | 採用。以降の土台 |
| v3 | タップ後に中間メニュー（要約/質問）を表示 | 「メニューを挟まず直接開きたい」で撤去 |
| v4 | シートを要約（上）/質問候補（下）の縦構成に | 「場所が狭い」指摘への対応 |
| v5 | FABの白基調をIndigo→Coral半透明グラデーションに | 「白はノートと同化する」指摘への対応 |
| v6 | シートタイトルの「（配下を含む）」文言を削除 | 表示の簡潔化 |

**教訓: 浮遊UIは「本文と区別できる色」「タップまでの手数最小」が生死を分ける。**

## 実装上の判断

- セクション分割は既存の `parseMarkdownBlocks()` の結果を再構成する純粋ロジック（`NoteSections.kt`）として実装し、UI非依存でテスト可能にした
- `MarkdownNoteContent` の `listState` はデフォルト引数とし、既存呼び出し（補記画面等）を無改修で維持した

- 会話履歴の永続化

## 追記

### 2026-07-20 — フォーカス周辺クイズ（Q&Aの吹き出しシートへの統合）

**背景**: 4択Q&Aは「ノート全体（実際は先頭1,200文字）から5問」だったが、2つの問題が同時に発覚した。(1) 超長いノートは冒頭からしか出題されない。(2) ML Kit genai-prompt beta2 の出力上限（256トークン強制）に5問は収まらず、**暗黙に途切れた分だけがパースされて動いていた**（`finishReason` 検知の導入で可視化）。

**判断**: 出題対象を「吹き出しがフォーカスしているセクションの周辺」に転換した。読んでいる場所に追従して出題されるためノートの長さが無関係になり、「読んだ部分を試す」という学習体験としても正しい形になる。さらに周辺本文の情報量に応じて形式と問題数を変え、小型モデルへ常に4択2問を強要しない。

| 論点 | 決定 | 理由 |
|------|------|------|
| 入口 | 吹き出しシート内「この部分でクイズ」。AIタブのQ&Aボタンは廃止 | 「吹き出し＝この部分へのAI操作の起点」の意味付けを強化。AIタブバッジは補記のみに |
| 周辺の定義 | セクションを核に前後ブロックを交互に追加して約1,200字（`surroundingContext`、純ロジック＋テスト） | セクション単位でなくブロック単位の拡張なら親子セクションの本文重複がない |
| 出題形式 | 周辺本文をローカル判定し、短い断片は○×2問、通常・コード主体は3択2問、情報量が十分なら4択1問 | AIによる事前分類を増やさず、素材量に対して無理のない想起負荷へ調整する |
| 出力の上限対策 | ○×と3択は解説なし、4択だけ短い解説を1文付ける。問題・選択肢にも文字数上限を指示 | ML Kitの出力上限内で応答を完結させ、`MAX_TOKENS`による全結果破棄を減らす |
| 形式判定 | コード比率45%以上は3択、本文180字未満または文シグナル2以下は○×、本文700字以上かつ文シグナル6以上は4択、それ以外は3択 | 同じ入力なら必ず同じ形式となる、低負荷でテスト可能な初期ルール。閾値は実機結果を見て調整する |
| クイズの寿命 | セクションチャットセッションに従属（新セッション開始・明示終了で破棄） | 別セクションの古いクイズが「開く」として残り続ける問題の防止 |

**試行と撤退 — 「もう2問」追い生成**: 2問では物足りない対策として、既出問題の除外リスト付き追い生成を実装。さらに素材拡張（2,400字）と一般知識出題の許可（解説に「一般知識: 」ラベル強制）まで試したが、実機での成功率が低く廃止した。**教訓: 小型モデルに「同じ素材から別の問題を作る」類の差分生成は難しい。要求は毎回独立・自己完結にする。**
