## 1. 背景・目的

ノートを開く導線が「ランダム表示」しかなく、「意図が反映されず無関係なノートが出る」という不満があった。一方で「毎回キーワードを入力するのは面倒」という場面もある。この両方に応えるため、**フォルダ選択を共通土台**にした新タブ「さがす🔎」を追加し、2つのモードを用意した。

- **🔎 キーワードモード**: 自然文（例:「習慣化について書いたやつ」）で入力すると、選択フォルダ配下からローカルLLM（Gemini Nano）が最も合う3件を選ぶ。
- **🎰 ランダムモード**: 入力不要。選択フォルダ配下からランダムに3件を引く（AI不使用）。

フォルダで探索範囲を人間が指定するため、日本語クエリのトークナイズ（分かち書き）に依存せず、「無関係が出る」不満が原理的に解消される。ランダムモードは当初アイデアの「条件付きガチャ（半ランダム）」に相当し、条件＝フォルダに置き換えたもの。

## 2. 設計の要点

### 2-1. なぜ「フォルダ選択」なのか

オンデバイスの Gemini Nano は埋め込み検索を持たず、全ノートを読ませて探すことはできない。既存の関連ノート機能（`RelatedNotesUseCase`）と同じく「候補を絞る → AIに選ばせる」の2段構えが現実解となる。この「候補を絞る」を**ユーザーによるフォルダ選択**が担うことで、日本語キーワード抽出という難所を回避した。

Vault は `_License_Exam` のような実フォルダと、`0000.Index` のようなルート直下のコード付きノートが混在するため、フォルダ列挙は名称からは導出できず、SAF による実フォルダ列挙が必要（下記 2-3）。

### 2-2. 40件カットの扱い（キーワードモード）

Nano に渡せる候補数には上限があるため、`SearchPickerUseCase` は候補が **40件を超えるときだけ**キーワード再現率カットで40件に絞る。これは「狙いのノートを候補に残す（再現率）」ためのもので、最終的な精度は Nano が担保する。40件以下のフォルダではカットは発生せず、全件を Nano に渡す。

再現率カットは**文字bigramの重なり数**で並べ替える方式を採用。日本語トークナイザ辞書を要さず、タイトルとクエリの部分的な文字並びの一致で「取りこぼさない」ことを狙う。精度でなく再現率が目的なので、粗い一致で十分。

### 2-3. AI非対応時のフォールバック

`checkAvailability()` が `Available` 以外、または Nano が候補外/空を返した場合は、キーワードカット結果（またはフォルダ先頭）の3件を素の検索結果として返す。UIには「AI非対応のため、キーワード一致で表示しています」等の注記を添える。端末がNano非対応でも機能が壊れない。

## 3. 実装内容

### 3-1. データ層（`NoteRepository.kt`）

既存の `collectNotes` はフォルダ構造を捨ててフラットに全mdを返すため、2メソッドを追加した。

| 要素 | 内容 |
|------|------|
| `NoteFolder(name, documentId)` | Vault直下フォルダの表現。配下探索の起点に使う |
| `listTopLevelFolders(...)` | Vault第一階層のフォルダのみを名前昇順で列挙（ドリルダウンなし） |
| `collectNotesInScope(..., scope)` | `scope=null`→ルート直下の.mdのみ（非再帰）／`scope=NoteFolder`→そのフォルダ配下を再帰BFS収集 |

### 3-2. UseCase層（`domain/SearchPickerUseCase.kt`・新規）

`RelatedNotesUseCase` の「絞る→Nano選定→`cleanAiTitle`でタイトルをノートに戻す」骨格をフォーク。起点を「現在ノート」から「自然文クエリ」へ、絞り込みをプレフィックスから「40超のみ発火するbigram再現率カット」へ差し替えた。`pick(query, scopeNotes): PickerResult` を公開。

### 3-3. プロンプト層（`ai/PromptBuilder.kt`）

`buildPickerPrompt(query, candidateTitles)` を追加。出力は関連ノートと同型（タイトルのみ・1行1件・説明なし）とし、既存パーサ（`cleanAiTitle`）をそのまま流用できるようにした。選定理由は今回は表示しない方針のため出力させない。

### 3-4. ViewModel層（`NoteViewModel.kt`）

`SearchState`（Idle/Loading/Success/Error）を追加。`NoteUiState` に `folders`／`selectedFolder`／`searchState` を追加。メソッドは `loadFolders`／`selectSearchFolder`／`searchByKeyword`／`pickRandomInScope`。ノートを開く導線は既存の `openNote(RelatedNote)` を再利用する。

### 3-5. UI層（`ui/SearchScreen.kt`・新規）

`SearchTab` を新規作成。上部にフォルダchips（横スクロール、先頭は「ルート直下」）、その下に自然文入力欄、`🎰 ランダム`／`🔎 さがす` の2ボタン、結果パネル（3件、既存 `RelatedNoteItem` を再利用しタップで開く）。

### 3-6. 配線（`ui/AppScaffold.kt`・`MainActivity.kt`）

`AppDestination` に `Search("search", "さがす", "🔎")` を追加（ノートの次）。`MainActivity` の `NavHost` に `composable("search")` を追加し、タブ表示時に `loadFolders` を呼び、結果タップで `openNote`→ノートタブへ遷移。

## 4. スコープの確定事項（設計セッションの結論）

| 項目 | 決定 |
|------|------|
| モード | キーワード／ランダムの2つ、いずれも**3件**（1件は既存ランダム機能と重複のため） |
| フォルダ階層 | **第一階層のみ**（ドリルダウンなし）。配下ノートは再帰で収集 |
| `_AI補記` フォルダ | 検索スコープに**出す**（除外ロジックを書かず実装を軽くする） |
| 選定理由の表示 | **なし**（タイトル3件のみ） |
| 40超の絞り込み | ランダムでなく bigram 再現率カット（Nanoが後段で精度担保） |

### 推奨確認項目（実機）

1. さがすタブ表示時にフォルダchips（`_AI補記` 含む）が列挙されること。
2. フォルダ選択→`🎰 ランダム`で配下から3件が引かれること（AI不要で動作）。
3. フォルダ選択→自然文入力→`🔎 さがす`で意図に沿った3件が出ること。
4. 40件超のフォルダで、狙いのノートが結果に残ること（再現率カットの妥当性）。
5. 結果タップでノートタブに遷移し本文が開くこと。
6. Nano非対応端末（またはDL前）でフォールバックの注記付きで結果が出ること。
